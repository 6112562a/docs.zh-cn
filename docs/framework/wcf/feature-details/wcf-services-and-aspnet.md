<span data-ttu-id="73d98-109">ASP.NET HTTP 运行库处理 ASP.NET 请求但不参与处理的请求发送到 WCF 服务，即使这些服务承载在同一 AppDomain 中按原样 ASP.NET 内容也是如此。</span><span class="sxs-lookup"><span data-stu-id="73d98-109">The ASP.NET HTTP runtime handles ASP.NET requests but does not participate in the processing of requests destined for WCF services, even though these services are hosted in the same AppDomain as is the ASP.NET content.</span></span> <span data-ttu-id="73d98-110">相反，WCF 服务模型会截获消息发送到 WCF 服务，并将其路由通过 WCF 传输协议/通道堆栈。</span><span class="sxs-lookup"><span data-stu-id="73d98-110">Instead, the WCF Service Model intercepts messages addressed to WCF services and routes them through the WCF transport/channel stack.</span></span>

<span data-ttu-id="73d98-111">并行模型的结果如下：</span><span class="sxs-lookup"><span data-stu-id="73d98-111">The results of the side-by-side model are as follows:</span></span>

  - <span data-ttu-id="73d98-127">ASP.NET 模拟：默认情况下，WCF 请求总是作为运行 IIS 进程标识，即使 ASP.NET 设置来启用模拟使用 System.Web 的\<identity impersonate ="true"/ > 配置选项。</span><span class="sxs-lookup"><span data-stu-id="73d98-127">ASP.NET impersonation: By default, WCF requests always runs as the IIS process identity, even if ASP.NET is set to enable impersonation using System.Web’s \<identity impersonate="true" /> configuration option.</span></span>

<span data-ttu-id="73d98-128">这些限制仅适用于 IIS 应用程序中承载的 WCF 服务。</span><span class="sxs-lookup"><span data-stu-id="73d98-128">These restrictions apply only to WCF services hosted in IIS application.</span></span> <span data-ttu-id="73d98-129">存在 WCF 不会影响 ASP.NET 内容的行为。</span><span class="sxs-lookup"><span data-stu-id="73d98-129">The behavior of ASP.NET content is not affected by the presence of WCF.</span></span>

## <a name="hosting-wcf-services-in-aspnet-compatibility-mode"></a><span data-ttu-id="73d98-136">在 ASP.NET 兼容模式下的托管 WCF 服务</span><span class="sxs-lookup"><span data-stu-id="73d98-136">Hosting WCF services in ASP.NET compatibility mode</span></span>

<span data-ttu-id="73d98-137">尽管 WCF 模型旨在跨宿主环境和传输行为一致，很多情况下应用程序不需要这种程度的灵活性。</span><span class="sxs-lookup"><span data-stu-id="73d98-137">Although the WCF model is designed to behave consistently across hosting environments and transports, there are often scenarios where an application does not require this degree of flexibility.</span></span> <span data-ttu-id="73d98-138">WCF 的 ASP.NET 兼容模式是适用于方案，不需要向 IIS 之外或通过除 HTTP 之外的协议进行通信的主机功能，但使用的所有 ASP.NET Web 应用程序平台的功能。</span><span class="sxs-lookup"><span data-stu-id="73d98-138">WCF’s ASP.NET compatibility mode is suitable for scenarios that do not require the ability to host outside of IIS or to communicate over protocols other than HTTP, but that use all of features of the ASP.NET Web application platform.</span></span>

<span data-ttu-id="73d98-139">与不同的是默认的并行配置中，其中 WCF 托管基础结构截获 WCF 消息，并将其路由到 HTTP 管道之外，在 ASP.NET 兼容模式中运行的 WCF 服务完全参与 ASP.NET HTTP 请求生命周期。</span><span class="sxs-lookup"><span data-stu-id="73d98-139">Unlike the default side-by-side configuration, where the WCF hosting infrastructure intercepts WCF messages and routes them out of the HTTP pipeline, WCF services running in ASP.NET Compatibility Mode participate fully in the ASP.NET HTTP request lifecycle.</span></span> <span data-ttu-id="73d98-140">在兼容性模式下，WCF 服务使用通过 HTTP 管道<xref:System.Web.IHttpHandler>实现中，类似于方法请求处理 ASPX 页和 ASMX Web 服务。</span><span class="sxs-lookup"><span data-stu-id="73d98-140">In compatibility mode, WCF services use the HTTP pipeline through an <xref:System.Web.IHttpHandler> implementation, similar to the way requests for ASPX pages and ASMX Web services are handled.</span></span> <span data-ttu-id="73d98-141">因此，WCF 的行为相同 ASMX 以下 ASP.NET 功能方面：</span><span class="sxs-lookup"><span data-stu-id="73d98-141">As a result, WCF behaves identically to ASMX with respect to the following ASP.NET features:</span></span>

- <span data-ttu-id="73d98-143">基于文件的授权：在 ASP.NET 兼容模式下运行的 WCF 服务可以通过将文件系统访问控制列表 (Acl) 附加到服务的.svc 文件安全。</span><span class="sxs-lookup"><span data-stu-id="73d98-143">File-based authorization: WCF services running in ASP.NET compatibility mode can be secure by attaching file system access control lists (ACLs) to the service’s .svc file.</span></span>
