# <a name="abstractions-abstract-types-and-interfaces"></a><span data-ttu-id="ee635-102">抽象（抽象类型和接口）</span><span class="sxs-lookup"><span data-stu-id="ee635-102">Abstractions (Abstract Types and Interfaces)</span></span>
<span data-ttu-id="ee635-103">抽象是一种描述协定但不提供协定的完整实现的类型。</span><span class="sxs-lookup"><span data-stu-id="ee635-103">An abstraction is a type that describes a contract but does not provide a full implementation of the contract.</span></span> <span data-ttu-id="ee635-104">抽象通常作为抽象类或接口实现，并带有一组明确定义的参考文档，该文档描述实现协定的类型的所需语义。</span><span class="sxs-lookup"><span data-stu-id="ee635-104">Abstractions are usually implemented as abstract classes or interfaces, and they come with a well-defined set of reference documentation describing the required semantics of the types implementing the contract.</span></span> <span data-ttu-id="ee635-105">.NET Framework 中一些最重要的抽象包括 <xref:System.IO.Stream>、<xref:System.Collections.Generic.IEnumerable%601> 和 <xref:System.Object>。</span><span class="sxs-lookup"><span data-stu-id="ee635-105">Some of the most important abstractions in the .NET Framework include <xref:System.IO.Stream>, <xref:System.Collections.Generic.IEnumerable%601>, and <xref:System.Object>.</span></span>  
  
 <span data-ttu-id="ee635-106">可以扩展框架，方法是：先实现一个支持抽象协定的具体类型，然后将该具体类型与使用（操作）该抽象的框架 API 配合使用。</span><span class="sxs-lookup"><span data-stu-id="ee635-106">You can extend frameworks by implementing a concrete type that supports the contract of an abstraction and using this concrete type with framework APIs consuming (operating on) the abstraction.</span></span>  
  
 <span data-ttu-id="ee635-107">能够经得起时间考验的有意义且有用的抽象很难设计。</span><span class="sxs-lookup"><span data-stu-id="ee635-107">A meaningful and useful abstraction that is able to withstand the test of time is very difficult to design.</span></span> <span data-ttu-id="ee635-108">主要的困难是设置合适的成员，不能多也不能少。</span><span class="sxs-lookup"><span data-stu-id="ee635-108">The main difficulty is getting the right set of members, no more and no fewer.</span></span> <span data-ttu-id="ee635-109">如果抽象具有太多成员，将很难或几乎不可能实现。</span><span class="sxs-lookup"><span data-stu-id="ee635-109">If an abstraction has too many members, it becomes difficult or even impossible to implement.</span></span> <span data-ttu-id="ee635-110">如果承诺功能的成员太少，那么在很多重要的场景中它就变得毫无用处。</span><span class="sxs-lookup"><span data-stu-id="ee635-110">If it has too few members for the promised functionality, it becomes useless in many interesting scenarios.</span></span>  
  
 <span data-ttu-id="ee635-111">框架中太多的抽象也会对框架的可用性产生负面影响。</span><span class="sxs-lookup"><span data-stu-id="ee635-111">Too many abstractions in a framework also negatively affect usability of the framework.</span></span> <span data-ttu-id="ee635-112">如果不理解抽象如何适应具体实现和抽象 API 操作的全局，通常很难理解抽象。</span><span class="sxs-lookup"><span data-stu-id="ee635-112">It is often quite difficult to understand an abstraction without understanding how it fits into the larger picture of the concrete implementations and the APIs operating on the abstraction.</span></span> <span data-ttu-id="ee635-113">此外，抽象的名称及其成员必然是抽象的，如果没有首先理解其使用的更广泛的背景，它们常常会变得更晦涩难懂。</span><span class="sxs-lookup"><span data-stu-id="ee635-113">Also, names of abstractions and their members are necessarily abstract, which often makes them cryptic and unapproachable without first understanding the broader context of their usage.</span></span>  
  
 <span data-ttu-id="ee635-114">但是，抽象提供了非常强大的可扩展性，其他可扩展性机制通常无法与其匹敌。</span><span class="sxs-lookup"><span data-stu-id="ee635-114">However, abstractions provide extremely powerful extensibility that the other extensibility mechanisms cannot often match.</span></span> <span data-ttu-id="ee635-115">它们是许多架构模式（例如插件、控制反转 (IoC)、管道等）的核心。</span><span class="sxs-lookup"><span data-stu-id="ee635-115">They are at the core of many architectural patterns, such as plug-ins, inversion of control (IoC), pipelines, and so on.</span></span> <span data-ttu-id="ee635-116">它们对于框架的可测试性也非常重要。</span><span class="sxs-lookup"><span data-stu-id="ee635-116">They are also extremely important for testability of frameworks.</span></span> <span data-ttu-id="ee635-117">良好的抽象能够为单元测试消除严重的依赖。</span><span class="sxs-lookup"><span data-stu-id="ee635-117">Good abstractions make it possible to stub out heavy dependencies for the purpose of unit testing.</span></span> <span data-ttu-id="ee635-118">总之，抽象让现代的面向对象的框架具有更受欢迎的丰富内涵。</span><span class="sxs-lookup"><span data-stu-id="ee635-118">In summary, abstractions are responsible for the sought-after richness of the modern object-oriented frameworks.</span></span>  
  
 <span data-ttu-id="ee635-119">**X 切忌** 提供抽象，除非通过开发几个使用抽象的具体实现和 API 来测试它们。</span><span class="sxs-lookup"><span data-stu-id="ee635-119">**X DO NOT** provide abstractions unless they are tested by developing several concrete implementations and APIs consuming the abstractions.</span></span>  
  
 <span data-ttu-id="ee635-120">**✓ 务必** 在设计抽象时，仔细选择使用抽象类还是接口。</span><span class="sxs-lookup"><span data-stu-id="ee635-120">**✓ DO** choose carefully between an abstract class and an interface when designing an abstraction.</span></span>  
  
 <span data-ttu-id="ee635-121">**✓ 考虑** 为抽象的具体实现提供参考测试。</span><span class="sxs-lookup"><span data-stu-id="ee635-121">**✓ CONSIDER** providing reference tests for concrete implementations of abstractions.</span></span> <span data-ttu-id="ee635-122">此类测试应允许用户测试其实现是否正确实现了协定。</span><span class="sxs-lookup"><span data-stu-id="ee635-122">Such tests should allow users to test whether their implementations correctly implement the contract.</span></span>
