## <a name="remoting"></a><span data-ttu-id="cb444-120">远程处理</span><span class="sxs-lookup"><span data-stu-id="cb444-120">Remoting</span></span>

<span data-ttu-id="cb444-121">.NET 远程处理被认为是存在问题的体系结构。</span><span class="sxs-lookup"><span data-stu-id="cb444-121">.NET Remoting was identified as a problematic architecture.</span></span> <span data-ttu-id="cb444-122">它用于进行跨 AppDomain 的通信，该通信现已不再受支持。</span><span class="sxs-lookup"><span data-stu-id="cb444-122">It's used for cross-AppDomain communication, which is no longer supported.</span></span> <span data-ttu-id="cb444-123">同样，远程处理也需要运行时支持，进行维护的成本较高。</span><span class="sxs-lookup"><span data-stu-id="cb444-123">Also, Remoting requires runtime support, which is expensive to maintain.</span></span> <span data-ttu-id="cb444-124">出于这些原因，.NET Core 不支持 .NET 远程处理，并且我们不计划在将来添加对它的支持。</span><span class="sxs-lookup"><span data-stu-id="cb444-124">For these reasons, .NET Remoting isn't supported on .NET Core, and we don't plan on adding support for it in the future.</span></span>

<span data-ttu-id="cb444-125">对于跨进程通信，可将进程间通信 (IPC) 机制视为远程处理的备用方案，如 <xref:System.IO.Pipes> 或 <xref:System.IO.MemoryMappedFiles.MemoryMappedFile> 类。</span><span class="sxs-lookup"><span data-stu-id="cb444-125">For communication across processes, consider inter-process communication (IPC) mechanisms as an alternative to Remoting, such as the <xref:System.IO.Pipes> or the <xref:System.IO.MemoryMappedFiles.MemoryMappedFile> class.</span></span>

<span data-ttu-id="cb444-126">对于跨计算机的通信，可将基于网络的解决方案用作备用方案。</span><span class="sxs-lookup"><span data-stu-id="cb444-126">Across machines, use a network-based solution as an alternative.</span></span> <span data-ttu-id="cb444-127">最好使用低开销纯文本协议，例如 HTTP。</span><span class="sxs-lookup"><span data-stu-id="cb444-127">Preferably, use a low-overhead plain text protocol, such as HTTP.</span></span> <span data-ttu-id="cb444-128">此处，ASP.NET Core 使用的 Web 服务器 [Kestrel Web 服务器](https://docs.microsoft.com/aspnet/core/fundamentals/servers/kestrel)是一个选择。</span><span class="sxs-lookup"><span data-stu-id="cb444-128">The [Kestrel web server](https://docs.microsoft.com/aspnet/core/fundamentals/servers/kestrel), the web server used by ASP.NET Core, is an option here.</span></span> <span data-ttu-id="cb444-129">也可考虑将 <xref:System.Net.Sockets> 用于基于网络的跨计算机的方案。</span><span class="sxs-lookup"><span data-stu-id="cb444-129">Also consider using <xref:System.Net.Sockets> for network-based, cross-machine scenarios.</span></span> <span data-ttu-id="cb444-130">有关更多选项，请参阅 [.NET 开放源代码开发人员项目：消息传送](https://github.com/Microsoft/dotnet/blob/master/dotnet-developer-projects.md#messaging)。</span><span class="sxs-lookup"><span data-stu-id="cb444-130">For more options, see [.NET Open Source Developer Projects: Messaging](https://github.com/Microsoft/dotnet/blob/master/dotnet-developer-projects.md#messaging).</span></span>

## <a name="code-access-security-cas"></a><span data-ttu-id="cb444-131">代码访问安全性 (CAS)</span><span class="sxs-lookup"><span data-stu-id="cb444-131">Code Access Security (CAS)</span></span>

<span data-ttu-id="cb444-132">沙盒依赖为托管应用程序或库使用或运行提供资源的运行时或框架进行限制，其[在 .NET Framework 上不受支持](~/docs/framework/misc/code-access-security.md)，因此在 .NET Core 上也不受支持。</span><span class="sxs-lookup"><span data-stu-id="cb444-132">Sandboxing, which relies on the runtime or the framework to constrain which resources a managed application or library uses or runs, [isn't supported on .NET Framework](~/docs/framework/misc/code-access-security.md) and therefore is also not supported on .NET Core.</span></span> <span data-ttu-id="cb444-133">.NET Framework 和运行时中存在太多发生特权提升以继续将 CAS 视为安全边界的情况。</span><span class="sxs-lookup"><span data-stu-id="cb444-133">There are too many cases in the .NET Framework and the runtime where an elevation of privileges occurs to continue treating CAS as a security boundary.</span></span> <span data-ttu-id="cb444-134">此外，CAS 使实现更加复杂，通常会对无意使用它的应用程序造成正确性-性能影响。</span><span class="sxs-lookup"><span data-stu-id="cb444-134">In addition, CAS makes the implementation more complicated and often has correctness-performance implications for applications that don't intend to use it.</span></span>
